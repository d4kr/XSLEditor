<objective>
Add Cut, Copy, Paste, and Select All commands to the Edit menu, delegating each to the currently active CodeArea (RichTextFX) in the focused editor tab.

Purpose: Users currently have NO Edit menu items at all (`<Menu text="Edit"/>` is empty). Mouse-driven clipboard access is missing, and accelerator hints are absent from the menu UI. This plan implements EDIT-10..13.

Output:
- New public method `EditorController.getActiveCodeArea(): Optional<CodeArea>` (re-usable for any future MenuBar action that targets the active editor).
- Four `@FXML` handlers in `MainController` that delegate to the active CodeArea using `ifPresent`.
- Four populated `<MenuItem>` entries inside the Edit menu in `main.fxml` with `Shortcut+X/C/V/A` accelerators.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| System clipboard ↔ editor | OS clipboard contents (arbitrary text from any application) flow into the editor on paste; editor selection flows out on cut/copy. Same trust level as keyboard input — no command execution surface. |
| User input → MenuBar action handlers | UI gesture (mouse click on MenuItem) triggers `@FXML` handlers in MainController. No untrusted data crosses this boundary; the action is a fixed delegation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Tampering | `CodeArea.paste()` — clipboard text injected into editor buffer | accept | Clipboard paste in a developer tool is functionally equivalent to keyboard typing. There is no code execution, script evaluation, or shell-out path between editor buffer and the XSLT/XSL-FO pipeline. RESEARCH.md security domain confirms ASVS V5 is not in scope. No mitigation required. |
| T-25-02 | Spoofing | MenuBar accelerator vs. RichTextFX native key handler (`Shortcut+C/X/V/A`) | accept | RichTextFX `GenericStyledAreaBehavior` consumes the key event when CodeArea is focused (issue #185); MenuBar accelerator fires only when CodeArea is NOT focused. Both paths invoke `CodeArea.cut/copy/paste/selectAll()` and produce the same result. The menu's accelerator label (`Shortcut+C` etc.) is therefore truthful UI — both paths give the user the same clipboard outcome. No mitigation required. |
| T-25-03 | Denial of Service | `getActiveCodeArea()` returning `null` instead of `Optional.empty()` causing NPE in handler | mitigate | Method signature returns `Optional<CodeArea>` (not nullable `CodeArea`). All four call sites use `.ifPresent(...)` — see Task 2 acceptance criteria grep checks. Manual smoke test step 7 explicitly verifies no-tab no-op behavior. |
| T-25-04 | Information Disclosure | `Edit > Copy` exposes editor selection to system clipboard, readable by any process | accept | Standard text-editor behavior expected by users. The XSLEditor is a single-user developer tool (CLAUDE.md: "no auth, no multi-user"); no privileged data is in scope. No mitigation required. |
| T-25-05 | Elevation of Privilege | FXML `onAction` handler injection via malicious FXML file | accept | `main.fxml` is a build-time bundled resource loaded from the JAR (`getClass().getResource(...)` pattern in XSLEditorApp). Users cannot supply external FXML. No external-FXML-loading path exists in the app. No mitigation required. |

ASVS L1 disposition: no high-severity threats. All `mitigate` items have concrete implementation references in the task acceptance criteria.
</threat_model>
