  <action>
Create the directory packaging/ at the repo root (it does not exist yet) and write the file
packaging/entitlements.plist with the following exact content — do not alter whitespace, comments,
or key ordering:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <!-- Required: JVM uses JIT compilation; Hardened Runtime blocks JIT by default -->
    <key>com.apple.security.cs.allow-jit</key>
    <true/>

    <!-- Required: JVM loads native libraries (JavaFX, FOP, Saxon JNI) -->
    <!-- Also the critical workaround for JDK-8358723 / JDK-8369477 when the  -->
    <!-- main launcher would otherwise load libjli.dylib without this entitlement -->
    <key>com.apple.security.cs.disable-library-validation</key>
    <true/>

    <!-- Recommended: JVM may need to map executable memory for bytecode -->
    <key>com.apple.security.cs.allow-unsigned-executable-memory</key>
    <true/>
</dict>
</plist>
```

Three entitlements are required:
- com.apple.security.cs.allow-jit — JVM JIT needs writable+executable memory; without this the
  app crashes at launch under Hardened Runtime
- com.apple.security.cs.disable-library-validation — JavaFX and FOP load unsigned native .dylib
  files; also the workaround for JDK-8358723 regression (main launcher signed without entitlements)
- com.apple.security.cs.allow-unsigned-executable-memory — Java bytecode interpretation needs
  executable memory

Do NOT add the com.apple.security.app-sandbox entitlement — it is incompatible with JVM apps and
would prevent the app from launching.
  </action>

  </verify>

--- STEP 5: Insert before jpackage (both jobs, identical) ---

      - name: Import signing certificate
        env:
          MACOS_CERTIFICATE: ${{ secrets.MACOS_CERTIFICATE }}
          MACOS_CERTIFICATE_PASSWORD: ${{ secrets.MACOS_CERTIFICATE_PASSWORD }}
          MACOS_KEYCHAIN_PASSWORD: ${{ secrets.MACOS_KEYCHAIN_PASSWORD }}
        run: |
          echo "$MACOS_CERTIFICATE" | base64 --decode > /tmp/certificate.p12
          security create-keychain -p "$MACOS_KEYCHAIN_PASSWORD" build.keychain
          security set-keychain-settings -lut 21600 build.keychain
          security unlock-keychain -p "$MACOS_KEYCHAIN_PASSWORD" build.keychain
          security import /tmp/certificate.p12 \
            -k build.keychain -f pkcs12 \
            -T /usr/bin/codesign -T /usr/bin/security \
            -P "$MACOS_CERTIFICATE_PASSWORD"
          security list-keychains -d user -s build.keychain login.keychain
          security set-key-partition-list -S apple-tool:,apple: \
            -k "$MACOS_KEYCHAIN_PASSWORD" build.keychain
          rm -f /tmp/certificate.p12

The `security set-key-partition-list` command (line 7 of the run block) is CRITICAL. Without it,
codesign shows a GUI dialog on the headless runner and the job hangs indefinitely.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Secret → CI runner env | MACOS_CERTIFICATE (base64 p12), MACOS_CERTIFICATE_PASSWORD, MACOS_KEYCHAIN_PASSWORD, MACOS_SIGNING_IDENTITY injected as env vars; GitHub encrypts at rest and masks in logs |
| /tmp/certificate.p12 on runner disk | Decoded p12 written to runner filesystem; ephemeral runner but still plaintext key material on disk |
| build.keychain on runner | Temporary keychain containing the private signing key; exists for the duration of the jpackage step |
| jpackage signing → DMG artifact | Signed DMG uploaded as GitHub artifact; artifact is the trust anchor for end-users |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-01 | Information Disclosure | /tmp/certificate.p12 on runner disk | mitigate | `rm -f /tmp/certificate.p12` immediately after `security import` (already in keychain setup sequence). GitHub runners are ephemeral but the file must not persist longer than necessary. |
| T-21-02 | Information Disclosure | MACOS_CERTIFICATE_PASSWORD in logs | accept | GitHub Actions automatically masks secrets referenced via `${{ secrets.* }}` in log output. Risk is low; accepted. |
| T-21-03 | Elevation of Privilege | Stolen signing certificate used to sign malicious binary | transfer | Certificate stored as GitHub Secret (encrypted at rest by GitHub). PKCS12 password (MACOS_CERTIFICATE_PASSWORD) is a second factor. Revocation is possible via Apple Developer portal. Risk transferred to GitHub's secret infrastructure. |
| T-21-04 | Tampering | CI workflow modified to exfiltrate MACOS_CERTIFICATE | mitigate | The secret is only available to runners executing this repository's workflows (not fork PRs). Branch protection rules on main prevent unauthorized workflow changes. ASVS L1 V4 — access control on secrets is scoped to repo. |
| T-21-05 | Denial of Service | build.keychain left on runner if job is cancelled | accept | Runner is ephemeral — keychain is destroyed with the VM. The `if: always()` Delete keychain step handles normal failure cases. No persistent risk. |
| T-21-06 | Spoofing | Wrong value in MACOS_SIGNING_IDENTITY (full string instead of name only) | accept | If incorrectly configured, jpackage fails with "no identity found" error (safe failure, not a security issue). Documented in RESEARCH.md Pitfall 4 and in Phase 23 SIGNING.md. |
</threat_model>
