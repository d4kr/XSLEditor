<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner ↔ Apple notary service | App-specific password and Apple ID cross this boundary as TLS-protected HTTPS over `notarytool`'s internal client; injected from GitHub secrets as env vars |
| GitHub Secrets store ↔ CI runner env | Three Apple secrets (`APPLE_ID`, `APPLE_TEAM_ID`, `APPLE_APP_SPECIFIC_PASSWORD`) are injected by GitHub Actions into the per-step `env:` block; auto-masked in logs |
| End-user macOS machine ↔ stapled DMG | Notarization ticket is embedded in the DMG file; Gatekeeper validates it offline using the embedded ticket plus the Apple root cert chain present on every Mac |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Information Disclosure | `APPLE_APP_SPECIFIC_PASSWORD` echoed in CI logs | mitigate | Inject as per-step `env:` (not via shell echo); GitHub Actions auto-masks secrets matching the configured value; never `echo` the variable in scripts (Task 1/2 actions do not echo it) |
| T-22-02 | Spoofing | Adversary submits malicious app under our credentials | mitigate | App-specific password is scoped to notarization only and revocable at appleid.apple.com without affecting the main Apple ID; main Apple ID password is never used in CI |
| T-22-03 | Denial of Service | Apple notary service hangs indefinitely | mitigate | `--timeout 300s` flag on `notarytool submit` (Pitfall 1); failed run logs the submission UUID for retrieval via `notarytool log <uuid>` |
| T-22-04 | Tampering | Stapled DMG is modified between staple step and upload-artifact, embedding a stale or invalid ticket | accept | Same job, same runner, sequential steps; GitHub Actions does not allow concurrent writes to the same `output/` directory within a job. Risk is negligible; no mitigation needed |
| T-22-05 | Elevation of Privilege | Compromised app-specific password lets an attacker notarize arbitrary binaries under our team | mitigate | App-specific password is single-purpose (notarization), stored only in GitHub Secrets, never written to disk on the runner; can be revoked at appleid.apple.com without rotating the main Apple ID password |
| T-22-06 | Repudiation | Workflow silently produces an un-notarized DMG when secrets are missing, and the release publishes a broken artifact | mitigate | Hard-fail guard at the top of each Notarize step exits 1 if any of the three Apple secrets is empty (per RESEARCH Open Question #3 recommendation); CI clearly reports incomplete artifact rather than publishing a bad one |
| T-22-07 | Information Disclosure | Notarization log returned by Apple may contain build paths or internal binary names that leak project structure | accept | The `notarytool log` command is only invoked manually for failure diagnosis (not in the automated CI path); logs are visible only to repo collaborators with Actions log access. Acceptable for an internal developer tool |
</threat_model>
