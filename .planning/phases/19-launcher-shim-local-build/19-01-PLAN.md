# Plan 19-01: Launcher Shim, Manifest Update & Build Verification

## Goal
Add `Launcher.java` so `java -jar XSLEditor-*.jar` works without "JavaFX runtime components are missing"; update `shadowJar` manifest to point to it; verify Saxon/FOP service files survive the fat JAR merge; confirm `-Pversion=X.Y.Z` stamps `version.properties` correctly.

## Requirements
- BUILD-01: `java -jar` launches app cleanly
- BUILD-02: `-Pversion=0.4.0` produces JAR reporting `0.4.0`
- BUILD-03: `META-INF/services/javax.xml.transform.TransformerFactory` present and correct after `mergeServiceFiles()`

## Background
JavaFX apps in a fat JAR fail `java -jar` if the main class extends `Application` — the JVM bootstrap runs before the JavaFX module is initialised. The fix is a plain (non-JavaFX) `Launcher` class that delegates to `Application.launch()`. `XSLEditorApp` stays unchanged; only the manifest entry point changes.

`mergeServiceFiles()` already exists in `shadowJar`, but correctness must be confirmed (Saxon registers two `TransformerFactory` impls; the merged file must list both). `-Pversion` already flows through `processResources.expand()` — this plan verifies it, not reimplements it.

## Tasks

### 1. Create `src/main/java/ch/ti/gagi/xsleditor/Launcher.java`
```java
package ch.ti.gagi.xsleditor;

import javafx.application.Application;

public class Launcher {
    public static void main(String[] args) {
        Application.launch(XSLEditorApp.class, args);
    }
}
```
**Why**: `Launcher` must call `Application.launch(XSLEditorApp.class, args)` — not `XSLEditorApp.main(args)`. If `XSLEditorApp.main` calls `launch(args)` (the inherited `Application.launch()`), delegating to it from another class still triggers the same "JavaFX runtime components are missing" JVM bootstrap failure. Only calling `Application.launch(Class, args)` from a plain non-Application class bypasses that check.

### 2. Update `build.gradle` — three changes
a. `application { mainClass }` → `'ch.ti.gagi.xsleditor.Launcher'`
b. `jar { manifest }` → `'Main-Class': 'ch.ti.gagi.xsleditor.Launcher'`
c. `shadowJar { manifest }` → `'Main-Class': 'ch.ti.gagi.xsleditor.Launcher'`

### 3. Build fat JAR and verify manifest
```bash
./gradlew shadowJar
unzip -p build/libs/XSLEditor-*.jar META-INF/MANIFEST.MF
# Expected: Main-Class: ch.ti.gagi.xsleditor.Launcher
```

### 4. Verify Saxon/FOP service file survives merge
```bash
unzip -p build/libs/XSLEditor-*.jar META-INF/services/javax.xml.transform.TransformerFactory
# Expected: both net.sf.saxon.TransformerFactoryImpl and org.apache.fop.configuration.FopFactory (or similar) listed
```
If the file is missing or truncated, add `mergeServiceFiles()` is already present — investigate shadow version conflict.

### 5. Smoke-test `java -jar`
```bash
java -jar build/libs/XSLEditor-*.jar &
# App window must appear. Kill after visual confirmation.
```
If "JavaFX runtime components are missing" still appears: confirm JDK has JavaFX (use Liberica JDK+FX 21 locally if needed, or `--module-path` workaround is NOT required with Launcher shim).

### 6. Harden `-Pversion` override in `build.gradle`
Replace the hardcoded `version = '0.3.0'` line with a safe override:
```groovy
version = project.findProperty('version') ?: '0.3.0'
```
This makes the override unconditional — Gradle 9 property resolution semantics vary; the `findProperty` guard is strictly safer regardless. Then verify:
```bash
./gradlew shadowJar -Pversion=0.4.0
unzip -p build/libs/XSLEditor-0.4.0.jar version.properties
# Must output: version=0.4.0
# If output still shows 0.3.0, the override is broken — stop and debug before proceeding.
```

### 7. Restore version to `0.3.0` in `build.gradle`
After test, reset `version = '0.3.0'` (CI will override it at build time via `-Pversion`).
